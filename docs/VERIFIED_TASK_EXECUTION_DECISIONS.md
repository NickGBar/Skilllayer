# Verified Task Execution — Design Decisions (ADR)

> **DESIGN ONLY.** ADR-style record of the choices behind
> [`VERIFIED_TASK_EXECUTION_DESIGN.md`](VERIFIED_TASK_EXECUTION_DESIGN.md).
> Each row is a decision that shaped the design; "Revisit when" states the
> signal that should reopen it.

| # | Decision | Chosen option | Alternatives | Reason | Revisit when |
|---|----------|---------------|--------------|--------|--------------|
| 1 | What kind of feature is VTE? | **Orchestrator over the three existing professional skills** (Model B) with a factored shared lifecycle (Model C seam) | A: new standalone skill; C: pure internal lifecycle; D: replace the three | Reuse-first; avoids three re-implementations; keeps existing skills' behavior unchanged | A fourth lifecycle stage is needed that no existing skill covers |
| 2 | Persistence location | **`.skilllayer/tasks/<task-id>/`**, parallel to existing stores | Single `tasks.json`; external DB; `.git/`-adjacent | Matches existing `.skilllayer/` convention; per-task isolation; no new infra | Task count grows so large that per-dir listing is slow |
| 3 | Persistence primitives | **Reuse `atomic_write_json/text` + `memory_lock`** | New writer; sqlite | Already crash-safe (temp+`os.replace`+fsync) and cross-process locked on POSIX | A store needs transactional multi-file commit beyond one lock |
| 4 | Source-of-truth order | **Repo > structured evidence > contract > checkpoint > summary > narrative** | Trust checkpoint first (faster resume) | Repo is the only thing that cannot lie; matches existing "evidence over claims" stance | Never — this is the core principle |
| 5 | Summary authority | **`summary.md` is generated, never read back** | Parse summary on resume | Same rule as existing `INDEX.md`; prevents prose drift becoming truth | Never |
| 6 | Baseline capture | **Bounded fingerprint** (HEAD + in-scope sha256 + test/config hashes + interpreter) | Full-repo hash; git-only; content snapshot | Enough to detect the four freshness classes without hashing the world or reading secrets | A freshness class appears that scope hashes cannot detect |
| 7 | File hashing implementation | **Reuse `build_watch_file_changes_artifacts` hashing** | New walker | Already gitignore-aware, skips `.venv`/`node_modules`/build, 1 MiB cap | Its skip rules stop matching VTE's scope needs |
| 8 | Git access | **Read-only, allowlisted git** (`build_git_status_artifacts`, `_run_git` rev-parse) | Full git wrapper with writes | VTE must never mutate git state; allowlist already forbids write ops | VTE ever needs to create a commit/branch (would need new consent) |
| 9 | Who edits source | **Host agent only; `executed_by="host_agent"`** | SkillLayer applies edits | SkillLayer's safety model forbids opaque source edits; keeps verification independent of authorship | Never |
| 10 | Test evidence | **Reuse `TestRunner`/`build_run_tests_artifacts`** incl. `tests_started`/`environment_error` | Parse exit code only | Already distinguishes "passed" from "never ran"; the core anti-false-success signal | Never |
| 11 | Consent granularity | **One consent per task lifecycle**, scoped to its dir; renewed for delete/scope-expand/opt-in-after-decline | Global consent; per-write consent | Balances "no hidden writes" with usability; one approval covers foreseeable writes of one task | Users report either too many prompts or surprise writes |
| 12 | Ephemeral mode | **Supported** (run without writing when persistence declined) | Require persistence to run at all | Preserves value even when the user won't persist; honest about lost handoff | Ephemeral mode causes confusion about what was/wasn't saved |
| 13 | Contract mutability | **`contract.json` write-once; scope changes are append-only `scope_amendments`** | Rewrite `allowed_paths` in place | Preserves an audit trail of originally-agreed vs. widened scope | Amendments become so common the original scope is meaningless |
| 14 | Result/checkpoint history | **Latest-wins + rotated `history[]`, `HISTORY_KEEP` bound** | Keep every attempt; keep only latest | Mirrors `save_context_snapshot`; bounded disk; retains recent audit trail | A compliance need requires full unbounded history |
| 15 | Scope violation severity | **Distinct classes; unresolved+unapproved caps verdict at `CHANGE_CONTRACT_VIOLATED`** even if tests pass | Warn only; hard-fail any extra file | Formatter churn ≠ forbidden-file edit; but real violations must block success | The class taxonomy proves too coarse or too strict in practice |
| 16 | Never auto-revert | **User edits during a task are recorded as drift, never reverted** | Auto-restore baseline | VTE must not destroy user/agent work; rollback is git's job, user-driven | Never |
| 17 | Freshness gate on success | **`FRESHNESS_UNKNOWN` where freshness is required → non-success (`VALIDATION_BLOCKED`)** | Assume current if unsure | "Unknown" must never masquerade as "fresh"; consistent with confidence-reducing stance | Never |
| 18 | Handoff portability | **Vendor-neutral checkpoint; no provider chain-of-thought, no prompts, no assumed transcript access** | Store agent-native session blob | Must work Claude Code↔Codex↔human with zero shared transcript | A universal cross-agent session format emerges |
| 19 | Handoff state convention | **Reuse `PURPOSE/OBJECTIVE/CONSTRAINTS/COMPLETED/NEXT STEP`** parsed by `build_resume_work_artifacts` | New handoff grammar | Resume skill already parses it; free reuse | The convention proves too rigid for richer handoffs |
| 20 | Secret handling in records | **Redact before persist; scanner runs first; unredactable field → refuse to persist + `VALIDATION_BLOCKED`** | Store raw and warn | "Any secret leakage into a persisted record is a release blocker" | Never (hard security line) |
| 21 | Stored path form | **Repo-relative only; strip absolute/home paths** | Store absolute paths for convenience | Privacy; portability across machines/agents | Never |
| 22 | Routing behavior | **Recommend automatically, never auto-start tracking; VTE predicate before `safe_code_change`** | Auto-start VTE on any code task; never recommend | Explicit-consent principle; VTE is the fuller surface but must not hijack every request | VTE recommendation fires too often/rarely in dogfood |
| 23 | Git requirement (MVP) | **Git repositories only; non-git → `VALIDATION_BLOCKED`** | Support non-git via file hashes alone | Baseline/freshness lean on HEAD; non-git verification is weaker and separable | Strong demand for non-git project verification |
| 24 | Verdict computation site | **Single pure mapping function** from the three skills' verdicts + gates | Inline verdict logic per call site | Testable, one place to enforce hard gates, no drift between MCP/CLI | Verdict logic must diverge per interface |
| 25 | Schema evolution | **`schema_version` int per file; tolerate extra keys; higher-major → `FRESHNESS_UNKNOWN`** | No versioning; strict schema | Forward/backward tolerance without silently misreading a newer record | A breaking format change requires a migration path |
| 26 | Corrupt record handling | **Refuse to trust, never overwrite, surface `record_corrupt`, leave file as evidence** | Reset to default silently | Same discipline as existing `MemoryStateCorruptError`/`TodosStoreCorruptError` | Never |
| 27 | `.gitignore` handling | **Suggest an entry, never edit `.gitignore`** | Auto-add ignore rule | Matches every existing workflow; no hidden file edits | Never |
| 28 | Release Readiness coupling | **Optional pre-checkpoint gate; its blockers become advisory risks, never silent upgrades** | Always run it; never run it | Release-framed tasks benefit; but it must not gate every task or mask a blocked change | Users want it mandatory for release tasks |

## Unresolved safety risks / assumptions needing prototypes

- **Redaction completeness (ADR 20)** — needs a prototype proving the secret
  scanner runs before *every* persisted field, or a design that stores zero free
  text. Highest-risk item.
- **Freshness precision (ADR 6, 17)** — needs designed drift cases to confirm
  scope-hash comparison catches the intended staleness without false STALE on
  harmless unrelated commits.
- **Verdict mapping fidelity (ADR 24)** — assumes the three skills' verdicts map
  cleanly onto VTE's eight; a prototype table test should confirm no gap makes a
  blocked change look validated.

## Foundation Milestone A — Finalized Decisions (Implemented)

> Unlike the rows above, this section records decisions that are **implemented**
> in `src/skilllayer/tasks/persistence.py` (VTE Foundation A), not merely
> designed. It resolves the three items Phase 1 of that milestone required —
> task ID format, write-once contract semantics, and the persistence safety
> policy — plus implementation-driven clarifications that came up while
> building. See [`VTE_PERSISTENCE_SECURITY.md`](VTE_PERSISTENCE_SECURITY.md)
> for the full trust-model writeup.

| # | Decision | Chosen option | Alternatives | Reason | Revisit when |
|---|----------|---------------|--------------|--------|--------------|
| F1 | Task ID format | `<UTC timestamp %Y%m%dT%H%M%SZ>-<slug ≤40 chars>-<8 hex chars>`, generated by `generate_task_id`, validated by a single `_TASK_ID_RE` fullmatch (lowercase-only charset, no `/`, `\`, `..`, or control chars) | UUID4 only; timestamp-only; caller-supplied free-form id | Sortable, human-legible, collision-resistant (checked against the filesystem before returning, retried on collision — never silently reused); lowercase-only charset makes case-collision structurally impossible | A multi-repo or cross-host id needs a stronger uniqueness guarantee than local filesystem collision-checking |
| F2 | Objective hint → slug safety | The raw `objective_hint` is scanned against the same high-confidence secret patterns as the redaction gate *before* slugification; any match discards the whole hint in favor of the generic `"task"` slug | Slugify first, then scan the slug; truncate only | A task_id becomes a permanent directory name — even a partial credential surviving lowercasing/charset-stripping into a slug is an unacceptable risk, so the check runs on the raw text, not the already-mangled slug | Never (hard security line, same as ADR 20) |
| F3 | Write-once contract enforcement | Existence of `contract.json` checked once before acquiring `memory_lock` (fast path, avoids taking the lock for an obviously-already-done case) and checked **again inside the lock** (race guard) before the atomic write; a second call returns `RECORD_ALREADY_EXISTS` and never touches the existing bytes | Check only outside the lock; use a separate "creation marker" file | The outside-lock check alone would allow two concurrent callers to both pass it and race on the write; the existing `atomic_write_json` primitive has no built-in create-exclusive mode, so the exists-check must be inside the same critical section as the write | `atomic_write_json`/`atomic_write_text` grow a native O_EXCL-style mode, which could replace the manual double-check |
| F4 | Persistence safety policy — 3 field classes | `SAFE_STRUCTURED` (exact shape/allowlist match; a known secret pattern embedded inside still rejects the whole value, never redacted in place), `REDACTABLE_TEXT` (known secret patterns reject; diagnostic PII redacts in place; residual high-entropy text rejects), `FORBIDDEN_FREE_TEXT` (unconditional reject, reserved for future fields) | Two classes (structured/free-text); per-field bespoke validators with no shared policy | Uniform, recursive, testable gate reused across every field in every record type; the three-way split cleanly separates "must match a shape" from "may contain redactable PII" from "must never be prose" | A field is needed that is genuinely free prose but must still be persisted (would need a fourth, carefully-scoped policy, not a `FORBIDDEN_FREE_TEXT` exception) |
| F5 | Rejection over redaction for structured fields | A `SAFE_STRUCTURED` value that matches a known secret pattern is rejected outright, never redacted in place | Redact the matched substring within the structured value and keep the rest | Redacting inside a path, enum, or hash would silently corrust its meaning (e.g. a path with a token substring redacted mid-string is no longer a valid path) — reject is the only safe response for a value whose exact bytes matter | Never |
| F6 | Low-confidence sensitive text | Any REDACTABLE_TEXT field containing a ≥32-char unbroken alnum/dash/underscore run mixing at least one letter and one digit, not matched by any known pattern, is rejected as `low_confidence_sensitive_token` (confidence=`"low"`) rather than accepted or redacted | Accept with a warning; redact the whole run | "Any low-confidence sensitive field must be rejected" — an unrecognized-but-secret-shaped token is exactly the case this milestone was built to catch, and false positives (rejecting something benign) are the safe failure direction | The false-positive rate in real usage proves too high (e.g. legitimate long identifiers routinely rejected) |
| F7 | Consent binding | `TaskConsent` carries `task_id` (exact match required) and a `project_root_fingerprint` (sha256 of the resolved root path, not the raw path) rather than the raw project path | Store the raw path on the consent object; bind only to task_id | Avoids ever comparing or logging a raw absolute path for consent validation, consistent with F-class privacy rules, while still making "changing the project requires new consent" enforceable | Never |
| F8 | Consent read/write asymmetry | Reads (`read_task_contract`/`read_task_result`/`read_task_checkpoint`) are never consent-gated; only the three write functions require a matching `TaskConsent` | Gate reads too, requiring a lighter "read consent" | Reads cannot mutate anything, so gating them adds friction with no safety benefit — matches the existing memory subsystem's own always-readable stance (e.g. `rehydrate_context`) | A future field in a record is itself sensitive enough that even reading it should require fresh authorization |
| F9 | Result/checkpoint state vocabulary | Extended the milestone's required 7 states (`CONSENT_REQUIRED`/`CONSENT_INVALID`/`WRITE_COMPLETED`/`WRITE_REJECTED`/`RECORD_ALREADY_EXISTS`/`RECORD_NOT_FOUND`/`PERSISTENCE_BLOCKED`) with one addition, `READ_COMPLETED`, for successful reads | Reuse `WRITE_COMPLETED` for successful reads too | A read that "writes nothing" reported as `WRITE_COMPLETED` would be misleading in logs/telemetry; the required 7 states are a minimum for the write/consent boundary, not an exhaustive enum | A stricter interpretation of the brief requires exactly those 7 states with no additions |
| F10 | Checkpoint versioning | `checkpoint_version` must be a positive int strictly greater than the existing checkpoint's version (read under the same operation, refusing the write with `checkpoint_version_not_monotonic` otherwise); an unreadable/corrupt existing checkpoint blocks the write rather than silently overwriting it | Allow any int; auto-increment server-side | Monotonicity makes "is this checkpoint newer" a trivial integer comparison for a future resume/freshness check, and refusing to overwrite an unreadable checkpoint preserves forensic evidence of the corruption | Auto-derived attempt/version numbers prove error-prone for callers in practice |
| F11 | Result attempt auto-derivation guard | When `attempt` is omitted, it is derived as `(existing result.json's attempt) + 1` — but only if the existing file reads back cleanly; if it exists but is corrupt/unreadable, the write is blocked (`existing_result_unreadable`) rather than silently defaulting to `attempt=1` | Default to `attempt=1` whenever the prior read fails, corrupt or missing alike | Silently resetting the attempt counter on a corrupt file could mask the corruption (same reasoning as the existing `.state.json`/`todos.json` corrupt-store handling in `memory/skilllayer_memory.py`) | Never |
| F12 | Lock granularity | Reused `memory_lock(project_root/.skilllayer)` — the existing, whole-store lock — rather than building a per-task lock | Add a second, task-scoped locking mechanism | "Do not create a second unrelated locking framework" (explicit instruction); accepted as a foundation-level limitation, documented in `VTE_PERSISTENCE_SECURITY.md` | Concurrent multi-task write contention in one repository becomes a measured problem in dogfooding |
| F13 | Entropy-heuristic false positives on structurally-verifiable identifiers | Exempted git SHAs (7–128 hex chars), sha1/256/384/512 digests, UUIDs, pip/npm-style package integrity hashes, and this system's own `task_id` shape from the low-confidence entropy fallback — checked only after every specific high-confidence reject pattern has already run and found nothing | Leave the heuristic as originally shipped (rejects these); broaden the exemption further to cover arbitrary long filenames/identifiers | Independent verification (negative-control testing) found git SHAs, SHA-256 hashes, UUIDs, task IDs, and package integrity hashes — all explicitly common, non-secret artifacts this system's own messaging needs to reference — were being rejected purely for length/mixed-alnum. Fixing this is a correctness fix, not a security weakening: the specific reject patterns are completely unaffected and still catch real secrets. Deliberately did NOT extend the exemption to arbitrary long filenames/opaque identifiers — those belong in `SAFE_STRUCTURED` path fields in this schema, not free text, and a fully general exemption would reopen the false-negative gap the heuristic exists to close | The false-positive/false-negative balance needs retuning after real dogfooding data, or a new common identifier shape (e.g. a different hash algorithm or ID scheme) needs its own exemption |
