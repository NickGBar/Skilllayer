# Verified Task Execution — Design Decisions (ADR)

> **DESIGN ONLY.** ADR-style record of the choices behind
> [`VERIFIED_TASK_EXECUTION_DESIGN.md`](VERIFIED_TASK_EXECUTION_DESIGN.md).
> Each row is a decision that shaped the design; "Revisit when" states the
> signal that should reopen it.

| # | Decision | Chosen option | Alternatives | Reason | Revisit when |
|---|----------|---------------|--------------|--------|--------------|
| 1 | What kind of feature is VTE? | **Orchestrator over the three existing professional skills** (Model B) with a factored shared lifecycle (Model C seam) | A: new standalone skill; C: pure internal lifecycle; D: replace the three | Reuse-first; avoids three re-implementations; keeps existing skills' behavior unchanged | A fourth lifecycle stage is needed that no existing skill covers |
| 2 | Persistence location | **`.skilllayer/tasks/<task-id>/`**, parallel to existing stores | Single `tasks.json`; external DB; `.git/`-adjacent | Matches existing `.skilllayer/` convention; per-task isolation; no new infra | Task count grows so large that per-dir listing is slow |
| 3 | Persistence primitives | **Reuse `atomic_write_json/text` + `memory_lock`** | New writer; sqlite | Already crash-safe (temp+`os.replace`+fsync) and cross-process locked on POSIX | A store needs transactional multi-file commit beyond one lock |
| 4 | Source-of-truth order | **Repo > structured evidence > contract > checkpoint > summary > narrative** | Trust checkpoint first (faster resume) | Repo is the only thing that cannot lie; matches existing "evidence over claims" stance | Never — this is the core principle |
| 5 | Summary authority | **`summary.md` is generated, never read back** | Parse summary on resume | Same rule as existing `INDEX.md`; prevents prose drift becoming truth | Never |
| 6 | Baseline capture | **Bounded fingerprint** (HEAD + in-scope sha256 + test/config hashes + interpreter) | Full-repo hash; git-only; content snapshot | Enough to detect the four freshness classes without hashing the world or reading secrets | A freshness class appears that scope hashes cannot detect |
| 7 | File hashing implementation | **Reuse `build_watch_file_changes_artifacts` hashing** | New walker | Already gitignore-aware, skips `.venv`/`node_modules`/build, 1 MiB cap | Its skip rules stop matching VTE's scope needs |
| 8 | Git access | **Read-only, allowlisted git** (`build_git_status_artifacts`, `_run_git` rev-parse) | Full git wrapper with writes | VTE must never mutate git state; allowlist already forbids write ops | VTE ever needs to create a commit/branch (would need new consent) |
| 9 | Who edits source | **Host agent only; `executed_by="host_agent"`** | SkillLayer applies edits | SkillLayer's safety model forbids opaque source edits; keeps verification independent of authorship | Never |
| 10 | Test evidence | **Reuse `TestRunner`/`build_run_tests_artifacts`** incl. `tests_started`/`environment_error` | Parse exit code only | Already distinguishes "passed" from "never ran"; the core anti-false-success signal | Never |
| 11 | Consent granularity | **One consent per task lifecycle**, scoped to its dir; renewed for delete/scope-expand/opt-in-after-decline | Global consent; per-write consent | Balances "no hidden writes" with usability; one approval covers foreseeable writes of one task | Users report either too many prompts or surprise writes |
| 12 | Ephemeral mode | **Supported** (run without writing when persistence declined) | Require persistence to run at all | Preserves value even when the user won't persist; honest about lost handoff | Ephemeral mode causes confusion about what was/wasn't saved |
| 13 | Contract mutability | **`contract.json` write-once; scope changes are append-only `scope_amendments`** | Rewrite `allowed_paths` in place | Preserves an audit trail of originally-agreed vs. widened scope | Amendments become so common the original scope is meaningless |
| 14 | Result/checkpoint history | **Latest-wins + rotated `history[]`, `HISTORY_KEEP` bound** | Keep every attempt; keep only latest | Mirrors `save_context_snapshot`; bounded disk; retains recent audit trail | A compliance need requires full unbounded history |
| 15 | Scope violation severity | **Distinct classes; unresolved+unapproved caps verdict at `CHANGE_CONTRACT_VIOLATED`** even if tests pass | Warn only; hard-fail any extra file | Formatter churn ≠ forbidden-file edit; but real violations must block success | The class taxonomy proves too coarse or too strict in practice |
| 16 | Never auto-revert | **User edits during a task are recorded as drift, never reverted** | Auto-restore baseline | VTE must not destroy user/agent work; rollback is git's job, user-driven | Never |
| B1 | Baseline storage | **Dedicated write-once `baseline.json`** | Put mutable facts in `contract.json` | Contract consent/scope stays immutable while factual evidence has a clear capture time | Evidence needs multiple snapshots; introduce a separately versioned history deliberately |
| B2 | Scope mode | **EXPLICIT and PREFIX only; CANDIDATE is not executable permission** | Infer candidate scope from request or diff | Prevents an ambiguous suggestion from widening source authority | A later user-approved planning layer provides deterministic candidates |
| B3 | Attribution | **Report observed changes neutrally; distinguish freshness from scope** | Claim all in-scope changes are agent changes | Git and fingerprints cannot prove authorship under concurrent edits | A provenance mechanism has independently verified evidence |
| B4 | Internal exclusion | **Ignore only exact current task-record paths** | Ignore all `.skilllayer/` | Other tasks and global state are relevant safety evidence | Record namespaces are independently authenticated and proven irrelevant |
| C1 | Checkpoint history | **Immutable per-sequence history plus atomic latest pointer** | Latest-only checkpoint; mutable audit log | A later resumer must validate predecessor order and detect tampering rather than trust the newest filename | Storage limits need a separately designed archival policy |
| C2 | Resume authority | **Fresh repository facts + Foundation B scope result override checkpoint claims** | Trust checkpoint narrative | A checkpoint can be stale or interrupted; the repository remains factual source of truth | Never |
| C3 | Interruption | **Unknown command/validation outcome requires confirmation and rerun** | Infer success from process start | Prevents false success after a terminated host or timed-out process | Never |
| C4 | Ownership | **Explicit short lease; no daemon heartbeat** | Permanent lock; background monitor | Blocks concurrent writers without leaving a permanent lock after process death | Explicit operations need longer leases in real use |
| 17 | Freshness gate on success | **`FRESHNESS_UNKNOWN` where freshness is required → non-success (`VALIDATION_BLOCKED`)** | Assume current if unsure | "Unknown" must never masquerade as "fresh"; consistent with confidence-reducing stance | Never |
| 18 | Handoff portability | **Vendor-neutral checkpoint; no provider chain-of-thought, no prompts, no assumed transcript access** | Store agent-native session blob | Must work Claude Code↔Codex↔human with zero shared transcript | A universal cross-agent session format emerges |
| 19 | Handoff state convention | **Reuse `PURPOSE/OBJECTIVE/CONSTRAINTS/COMPLETED/NEXT STEP`** parsed by `build_resume_work_artifacts` | New handoff grammar | Resume skill already parses it; free reuse | The convention proves too rigid for richer handoffs |
| 20 | Secret handling in records | **Redact before persist; scanner runs first; unredactable field → refuse to persist + `VALIDATION_BLOCKED`** | Store raw and warn | "Any secret leakage into a persisted record is a release blocker" | Never (hard security line) |
| 21 | Stored path form | **Repo-relative only; strip absolute/home paths** | Store absolute paths for convenience | Privacy; portability across machines/agents | Never |
| 22 | Routing behavior | **Recommend automatically, never auto-start tracking; VTE predicate before `safe_code_change`** | Auto-start VTE on any code task; never recommend | Explicit-consent principle; VTE is the fuller surface but must not hijack every request | VTE recommendation fires too often/rarely in dogfood |
| 23 | Git requirement (MVP) | **Git repositories only; non-git → `VALIDATION_BLOCKED`** | Support non-git via file hashes alone | Baseline/freshness lean on HEAD; non-git verification is weaker and separable | Strong demand for non-git project verification |
| 24 | Verdict computation site | **Single pure mapping function** from the three skills' verdicts + gates | Inline verdict logic per call site | Testable, one place to enforce hard gates, no drift between MCP/CLI | Verdict logic must diverge per interface |
| 25 | Schema evolution | **`schema_version` int per file; tolerate extra keys; higher-major → `FRESHNESS_UNKNOWN`** | No versioning; strict schema | Forward/backward tolerance without silently misreading a newer record | A breaking format change requires a migration path |
| 26 | Corrupt record handling | **Refuse to trust, never overwrite, surface `record_corrupt`, leave file as evidence** | Reset to default silently | Same discipline as existing `MemoryStateCorruptError`/`TodosStoreCorruptError` | Never |
| 27 | `.gitignore` handling | **Suggest an entry, never edit `.gitignore`** | Auto-add ignore rule | Matches every existing workflow; no hidden file edits | Never |
| 28 | Release Readiness coupling | **Optional pre-checkpoint gate; its blockers become advisory risks, never silent upgrades** | Always run it; never run it | Release-framed tasks benefit; but it must not gate every task or mask a blocked change | Users want it mandatory for release tasks |

| No unnecessary architecture expansion | ADR 2, 3; everything in §18 deferred |

## Unresolved safety risks / assumptions needing prototypes

- **Redaction completeness (ADR 20)** — needs a prototype proving the secret
  scanner runs before *every* persisted field, or a design that stores zero free
  text. Highest-risk item.
- **Freshness precision (ADR 6, 17)** — needs designed drift cases to confirm
  scope-hash comparison catches the intended staleness without false STALE on
  harmless unrelated commits.
- **Verdict mapping fidelity (ADR 24)** — assumes the three skills' verdicts map
  cleanly onto VTE's eight; a prototype table test should confirm no gap makes a
  blocked change look validated.

## Foundation Milestone A — Finalized Decisions (Implemented)

> Unlike the rows above, this section records decisions that are **implemented**
> in `src/skilllayer/tasks/persistence.py` (VTE Foundation A), not merely
> designed. It resolves the three items Phase 1 of that milestone required —
> task ID format, write-once contract semantics, and the persistence safety
> policy — plus implementation-driven clarifications that came up while
> building. See [`VTE_PERSISTENCE_SECURITY.md`](VTE_PERSISTENCE_SECURITY.md)
> for the full trust-model writeup.

| # | Decision | Chosen option | Alternatives | Reason | Revisit when |
|---|----------|---------------|--------------|--------|--------------|
| F1 | Task ID format | `<UTC timestamp %Y%m%dT%H%M%SZ>-<slug ≤40 chars>-<8 hex chars>`, generated by `generate_task_id`, validated by a single `_TASK_ID_RE` fullmatch (lowercase-only charset, no `/`, `\`, `..`, or control chars) | UUID4 only; timestamp-only; caller-supplied free-form id | Sortable, human-legible, collision-resistant (checked against the filesystem before returning, retried on collision — never silently reused); lowercase-only charset makes case-collision structurally impossible | A multi-repo or cross-host id needs a stronger uniqueness guarantee than local filesystem collision-checking |
| F2 | Objective hint → slug safety | The raw `objective_hint` is scanned against the same high-confidence secret patterns as the redaction gate *before* slugification; any match discards the whole hint in favor of the generic `"task"` slug | Slugify first, then scan the slug; truncate only | A task_id becomes a permanent directory name — even a partial credential surviving lowercasing/charset-stripping into a slug is an unacceptable risk, so the check runs on the raw text, not the already-mangled slug | Never (hard security line, same as ADR 20) |
| F3 | Write-once contract enforcement | Existence of `contract.json` checked once before acquiring `memory_lock` (fast path, avoids taking the lock for an obviously-already-done case) and checked **again inside the lock** (race guard) before the atomic write; a second call returns `RECORD_ALREADY_EXISTS` and never touches the existing bytes | Check only outside the lock; use a separate "creation marker" file | The outside-lock check alone would allow two concurrent callers to both pass it and race on the write; the existing `atomic_write_json` primitive has no built-in create-exclusive mode, so the exists-check must be inside the same critical section as the write | `atomic_write_json`/`atomic_write_text` grow a native O_EXCL-style mode, which could replace the manual double-check |
| F4 | Persistence safety policy — 3 field classes | `SAFE_STRUCTURED` (exact shape/allowlist match; a known secret pattern embedded inside still rejects the whole value, never redacted in place), `REDACTABLE_TEXT` (known secret patterns reject; diagnostic PII redacts in place; residual high-entropy text rejects), `FORBIDDEN_FREE_TEXT` (unconditional reject, reserved for future fields) | Two classes (structured/free-text); per-field bespoke validators with no shared policy | Uniform, recursive, testable gate reused across every field in every record type; the three-way split cleanly separates "must match a shape" from "may contain redactable PII" from "must never be prose" | A field is needed that is genuinely free prose but must still be persisted (would need a fourth, carefully-scoped policy, not a `FORBIDDEN_FREE_TEXT` exception) |
| F5 | Rejection over redaction for structured fields | A `SAFE_STRUCTURED` value that matches a known secret pattern is rejected outright, never redacted in place | Redact the matched substring within the structured value and keep the rest | Redacting inside a path, enum, or hash would silently corrust its meaning (e.g. a path with a token substring redacted mid-string is no longer a valid path) — reject is the only safe response for a value whose exact bytes matter | Never |
| F6 | Low-confidence sensitive text | Any REDACTABLE_TEXT field containing a ≥32-char unbroken alnum/dash/underscore run mixing at least one letter and one digit, not matched by any known pattern, is rejected as `low_confidence_sensitive_token` (confidence=`"low"`) rather than accepted or redacted | Accept with a warning; redact the whole run | "Any low-confidence sensitive field must be rejected" — an unrecognized-but-secret-shaped token is exactly the case this milestone was built to catch, and false positives (rejecting something benign) are the safe failure direction | The false-positive rate in real usage proves too high (e.g. legitimate long identifiers routinely rejected) |
| F7 | Consent binding | `TaskConsent` carries `task_id` (exact match required) and a `project_root_fingerprint` (sha256 of the resolved root path, not the raw path) rather than the raw project path | Store the raw path on the consent object; bind only to task_id | Avoids ever comparing or logging a raw absolute path for consent validation, consistent with F-class privacy rules, while still making "changing the project requires new consent" enforceable | Never |
| F8 | Consent read/write asymmetry | Reads (`read_task_contract`/`read_task_result`/`read_task_checkpoint`) are never consent-gated; only the three write functions require a matching `TaskConsent` | Gate reads too, requiring a lighter "read consent" | Reads cannot mutate anything, so gating them adds friction with no safety benefit — matches the existing memory subsystem's own always-readable stance (e.g. `rehydrate_context`) | A future field in a record is itself sensitive enough that even reading it should require fresh authorization |
| F9 | Result/checkpoint state vocabulary | Extended the milestone's required 7 states (`CONSENT_REQUIRED`/`CONSENT_INVALID`/`WRITE_COMPLETED`/`WRITE_REJECTED`/`RECORD_ALREADY_EXISTS`/`RECORD_NOT_FOUND`/`PERSISTENCE_BLOCKED`) with one addition, `READ_COMPLETED`, for successful reads | Reuse `WRITE_COMPLETED` for successful reads too | A read that "writes nothing" reported as `WRITE_COMPLETED` would be misleading in logs/telemetry; the required 7 states are a minimum for the write/consent boundary, not an exhaustive enum | A stricter interpretation of the brief requires exactly those 7 states with no additions |
| F10 | Checkpoint versioning | `checkpoint_version` must be a positive int strictly greater than the existing checkpoint's version (read under the same operation, refusing the write with `checkpoint_version_not_monotonic` otherwise); an unreadable/corrupt existing checkpoint blocks the write rather than silently overwriting it | Allow any int; auto-increment server-side | Monotonicity makes "is this checkpoint newer" a trivial integer comparison for a future resume/freshness check, and refusing to overwrite an unreadable checkpoint preserves forensic evidence of the corruption | Auto-derived attempt/version numbers prove error-prone for callers in practice |
| F11 | Result attempt auto-derivation guard | When `attempt` is omitted, it is derived as `(existing result.json's attempt) + 1` — but only if the existing file reads back cleanly; if it exists but is corrupt/unreadable, the write is blocked (`existing_result_unreadable`) rather than silently defaulting to `attempt=1` | Default to `attempt=1` whenever the prior read fails, corrupt or missing alike | Silently resetting the attempt counter on a corrupt file could mask the corruption (same reasoning as the existing `.state.json`/`todos.json` corrupt-store handling in `memory/skilllayer_memory.py`) | Never |
| F12 | Lock granularity | Reused `memory_lock(project_root/.skilllayer)` — the existing, whole-store lock — rather than building a per-task lock | Add a second, task-scoped locking mechanism | "Do not create a second unrelated locking framework" (explicit instruction); accepted as a foundation-level limitation, documented in `VTE_PERSISTENCE_SECURITY.md` | Concurrent multi-task write contention in one repository becomes a measured problem in dogfooding |
| F13 | Entropy-heuristic false positives on structurally-verifiable identifiers | Exempted git SHAs (7–128 hex chars), sha1/256/384/512 digests, UUIDs, pip/npm-style package integrity hashes, and this system's own `task_id` shape from the low-confidence entropy fallback — checked only after every specific high-confidence reject pattern has already run and found nothing | Leave the heuristic as originally shipped (rejects these); broaden the exemption further to cover arbitrary long filenames/identifiers | Independent verification (negative-control testing) found git SHAs, SHA-256 hashes, UUIDs, task IDs, and package integrity hashes — all explicitly common, non-secret artifacts this system's own messaging needs to reference — were being rejected purely for length/mixed-alnum. Fixing this is a correctness fix, not a security weakening: the specific reject patterns are completely unaffected and still catch real secrets. Deliberately did NOT extend the exemption to arbitrary long filenames/opaque identifiers — those belong in `SAFE_STRUCTURED` path fields in this schema, not free text, and a fully general exemption would reopen the false-negative gap the heuristic exists to close | The false-positive/false-negative balance needs retuning after real dogfooding data, or a new common identifier shape (e.g. a different hash algorithm or ID scheme) needs its own exemption |

## Foundation Milestone D — Finalized Decisions (Implemented)

> Records decisions made building `src/skilllayer/tasks/orchestrator.py` (VTE
> Foundation D), which connects Foundations A/B/C without modifying any of
> their source. See
> [`VTE_FOUNDATION_D_ORCHESTRATOR.md`](VTE_FOUNDATION_D_ORCHESTRATOR.md) for
> the full write-up.

| # | Decision | Chosen option | Alternatives | Reason | Revisit when |
|---|----------|---------------|--------------|--------|--------------|
| D1 | `create_task` takes `task_id` explicitly | Caller generates `task_id` first (via `persistence.generate_task_id`, exactly as Foundation A's own `create_task_contract` already requires) and passes it in, rather than `create_task` minting one internally | Auto-generate `task_id` inside `create_task` | A `TaskConsent` must be bound to a specific `task_id` *before* the call — auto-generating internally creates an unresolvable chicken-and-egg problem (the caller cannot construct valid consent for an id it doesn't know yet). Discovered by actually running the happy-path scenario, not by review | Foundation A ever supports a consent object valid for "any task_id this specific call creates" |
| D2 | `create_task` idempotency via a caller-supplied key, not `task_id` reuse | A bounded `.skilllayer/tasks/.idempotency_index.json` maps `idempotency_key -> task_id`; a retry with the same key returns the *original* task regardless of what (possibly fresh, different) `task_id`/consent the retry call itself supplied | Require the caller to pass the same `task_id` back on retry | A genuine retry commonly cannot avoid generating a fresh `task_id` (D1), so `task_id` cannot double as the idempotency key; a caller-chosen stable key (e.g. their own request id) is the only thing that can be reused across a retry when the id itself can't be | Never |
| D3 | Checkpoint idempotency re-checks the *current latest* sequence, not `latest.sequence + 1` | Before advancing to a new sequence, `checkpoint_task` checks whether the current latest checkpoint's id already equals what `(task_id, latest.sequence, idempotency_key)` would derive — if so, it's a retry, return the existing one unchanged | Derive `checkpoint_id` at the pre-computed next sequence unconditionally | Found via testing: sequence naturally advances once the first call already succeeded, so blindly deriving at `latest.sequence + 1` on a "retry" silently mints a genuinely NEW checkpoint every time instead of recognizing the prior one | Never |
| D4 | `acquire_ownership` vs `reclaim_ownership` — two functions, not one | `acquire_ownership` is the one-time initial lease tied to the `BASELINE_READY -> READY_TO_START` transition; `reclaim_ownership` is a separate operation, valid from any non-terminal state, for re-acquiring an expired lease later without re-running (or being blocked by) the one-time state transition | One `acquire_ownership` function reusable at any point | Found via testing: gating re-acquisition on `current_state == BASELINE_READY` made a legitimately expired lease during `RUNNING`/`INTERRUPTED` unreclaimable, directly contradicting the explicit requirement that expired ownership "may be reclaimed through an explicit operation" | Foundation C grows a native reclaim-vs-acquire distinction, which this could then delegate to |
| D5 | Self-ownership is reinterpreted, not re-derived, in `resume_task` | When Foundation C's `assess_resume` blocks solely on `active_task_owner` and the caller is independently verified (via `_ownership_is_active`) to be that same active owner, the verdict is treated as `RESUME_SAFE_WITH_CONFIRMATION` rather than `RESUME_BLOCKED` — the raw assessment is preserved, only the reasons/verdict fields used for this decision are annotated | Modify Foundation C's `assess_resume` to accept an `owner_instance_id` and skip the check for a matching owner | `assess_resume` is conservative by design (no third party may ever assume ownership from an active lease) and Foundation A/B/C are meant to stay unmodified by D per the milestone brief; reinterpreting locally, without skipping the deeper checks Foundation C's early return omits (they still run for real in the next `validate_task` call), preserves the conservative default for everyone except the one case the orchestrator has already independently verified is safe | Foundation C adds owner-aware resume assessment natively |
| D6 | `final_result` persisted as its own evidence record, separate from `result.json` | `finalize_task` writes the exact dict it returns to `evidence/final_result.json` (via the same `_write_evidence` helper used for scope validation) in addition to Foundation A's `write_task_result` | Reconstruct the orchestrator's `final_result` shape from `result.json` on a repeat call | `result.json`'s schema (Foundation A, `attempt`/`verdict`/`verdict_reasons`) and the orchestrator's `final_result` shape (`final_state`/`blockers`/`allowed_changes`/...) are different schemas for different audiences; found via testing that reading back `result.json` for an idempotent retry returned a differently-shaped dict than the first call, breaking "repeated finalize returns the same result" in a literal sense | Never — the two schemas serve genuinely different purposes (shared cross-workflow result vs. this module's own detailed verdict) |
| D7 | Orchestrator-local paths (`state.json`, `transitions/`, `evidence/`) are computed as children of `task_dir`, never added to `persistence.TaskRecordPaths` | Foundation A/B/C's dataclass and modules are left completely unmodified; Foundation D derives its own paths from the already-returned, already symlink-checked `task_dir`, with its own additional per-path symlink checks mirroring Foundation C's `checkpoints_dir` pattern | Extend `TaskRecordPaths` with new fields for Foundation D's paths | Minimizes the diff against already-reviewed/merged code; any path that is a child of an already-validated `task_dir` is automatically confined, so no new escape vector is introduced by computing it externally | A future foundation needs these paths from *outside* the `tasks` subpackage, at which point centralizing them in `TaskRecordPaths` becomes worth the shared-code churn |
| D8 | `BLOCKED` recovers only through a fresh `validate_task`, never directly to `finalize_task` | `finalize_task`'s only valid predecessor state is `VALIDATING`; `BLOCKED` must re-enter via `validate_task` (which re-derives scope fresh) before any further finalize attempt | Allow `finalize_task` directly from `BLOCKED`, re-deriving scope internally on the way in | Forces every finalize attempt to be immediately preceded by a fresh, explicit validation step in the transition history — an auditor reading the transition log never has to wonder whether a `BLOCKED -> COMPLETED` finalize silently re-validated internally | Never |

## Foundation D risks resolved during verification (not merely designed)

- **D1–D2** were found by actually running the dogfood scenario end-to-end,
  not by code review — the chicken-and-egg consent problem only surfaces
  once you try to call the functions in the order a real caller would.
- **D3–D4** were found the same way, via the required adversarial tests
  (repeated-checkpoint idempotency, expired-ownership reclaim) actually
  failing on first implementation.
- **D5** required understanding Foundation C's own conservative-by-design
  ownership check well enough to distinguish "genuinely blocked" from
  "blocked only because assess_resume can't see who's asking."

## Foundation Milestone E — Finalized Decisions (Implemented)

> Records decisions made building the public product surface
> (`skilllayer.tasks.public_api`/`receipt`/`interventions`,
> `skilllayer_vte_*` MCP tools, and the `verified_task_execution`
> professional skill catalog entry) on top of the unmodified Foundation D

| # | Decision | Chosen option | Alternatives | Reason | Revisit when |
|---|----------|---------------|--------------|--------|--------------|
| E1 | Consent is granted internally on every call after `vte_start`, gated only once at task creation | `vte_start` is the only tool requiring an explicit `persist_consent=True`; every later tool (`vte_checkpoint`/`vte_resume`/`vte_finalize`) constructs a fresh `TaskConsent` via `grant_task_consent(project_root, task_id)` for the already-created task | Require `persist_consent` (or an equivalent) on every VTE tool call | `TaskConsent` is a call-time assertion bound to `(task_id, project_root)`, not a stored secret — Foundation A already reconstructs it the same way internally; re-prompting for consent on every checkpoint would add friction without adding any real gate, since the task's existence already reflects the user's one deliberate consent decision | Foundation A ever adds a genuinely revocable, persisted consent grant |
| E2 | `vte_checkpoint` builds Foundation C step/evidence records from plain string labels | Caller-supplied `completed_steps`/`active_step`/`remaining_steps` are plain strings; this module slugifies them into valid `step_id`s and writes one bounded `checkpoint_claim_<seq>.json` evidence record per call, referenced by every completed step in that call | Require the caller to construct Foundation C's `_valid_step`/`_valid_evidence_ref` shapes directly | Milestone requirement: "no need to construct evidence digests manually." A completed step with no evidence is rejected by Foundation C itself (`completed_step_missing_evidence`); always attaching real evidence means a completion claim is only ever rejected for a genuine schema problem, never silently dropped | Foundation C adds a native plain-label checkpoint constructor |
| E3 | `vte_resume`'s confirmation gate is driven by `resume_task`'s own returned state, not by the raw `assess_task_resume` verdict | The wrapper always calls `resume_task` and reacts to whatever state comes back (`CONFIRMATION_REQUIRED`/`TASK_BLOCKED`/success) | Pre-branch on `assess_task_resume`'s raw `resume_verdict` before deciding whether to call `resume_task` | **Found via testing**: `assess_task_resume` has no `owner_instance_id` and cannot apply the self-ownership reinterpretation only `resume_task` can do (see Foundation D's D5); pre-branching on the raw verdict caused a same-owner interrupted task to skip the confirmation flow entirely and surface `resume_task`'s internal `CONFIRMATION_REQUIRED` state as a bare error instead of a proper confirmation-token response | Never |
| E4 | A supplied `confirmation_token` is always consumed, even when the caller also passes `confirmed=True` | `token_consumed = _consume_confirmation(...) if confirmation_token is not None else False; effective_confirmed = confirmed or token_consumed` — the token check always runs when a token is present | `effective_confirmed = confirmed or _consume_confirmation(...)` (short-circuits, skipping consumption whenever `confirmed=True` is already set) | **Found via testing** (`test_confirmation_token_is_single_use`): the short-circuiting `or` left a token's `consumed` flag `False` forever whenever the caller passed `confirmed=True` alongside it, so that same token could later be replayed to bypass confirmation on an unrelated, later interruption of the same task — a genuine single-use violation, not a cosmetic bug | Never |
| E5 | `vte_finalize`'s state precheck admits `COMPLETED`/`FAILED`, not only `RUNNING`/`VALIDATING`/`BLOCKED` | The precheck rejects only states where `finalize_task` itself has no valid entry point at all; `COMPLETED`/`FAILED` are let through because `finalize_task` has its own idempotent-retry path for exactly those two states | Reject any state other than `RUNNING`/`VALIDATING`/`BLOCKED` before ever calling `finalize_task` | **Found via testing** (`test_finalize_is_idempotent_on_retry`): a second `vte_finalize` call on an already-`COMPLETED` task was rejected by this wrapper's own precheck before it ever reached `finalize_task`'s working idempotent-retry logic, turning a documented Foundation D guarantee into a wrapper-level regression | Never |
| E6 | `receipt.py`'s `SKILL_NAME` constant is a literal string equal to the catalog's name, not re-derived from it | `SKILL_NAME = "verified_task_execution"`, hand-kept identical to `config.defaults.VERIFIED_TASK_EXECUTION_SKILL["name"]` (a test asserts equality) | Have `receipt.py` import the name from `config.defaults` directly | **Found via testing**: the two were written independently (`"verified-task-execution"` vs `"verified_task_execution"`) and diverged; a direct import would remove the drift risk entirely, but was intentionally deferred to avoid `receipt.py` (a `tasks`-subpackage module) importing from `config`, a layering direction not used elsewhere in `tasks/` — the equality is instead pinned by a dedicated test | If another skill-identified module is added, centralize the name constant and revisit this layering call |
| E7 | The `verified_task_execution` professional skill is cataloged (via `skilllayer_list_skills` and `diagnostics.py`) but deliberately NOT wired into `SkillRouter`'s automatic `task_type` classification | `safe_code_change`/`release_readiness`/`resume_project_work` all auto-route free text to an internal workflow executed via `skilllayer_run`; VTE's six tools are never invoked that way — every operation is an explicit `vte_*` tool call the agent chooses to make | Add a `verified_task_execution` task_type to `SkillRouter`, routed to a workflow that calls the VTE tools internally | Automatic routing to a multi-step, ownership-acquiring, consent-gated lifecycle from free-text classification would make the *first* write (`vte_start`) a side effect of a guess rather than a deliberate choice, undermining the explicit-consent design this whole surface is built around; this mirrors Foundation D's own "no router activation" boundary | The product wants one-shot "just do it as a verified task" invocation without an agent explicitly sequencing the six tools itself |

## Foundation Milestone E risks resolved during verification (not merely designed)

- **E3–E5** were all found by writing and running the required adversarial
  tests, not by review: none of the three would have been obvious from
  reading `public_api.py` in isolation, since each depends on how a
  specific Foundation D return shape (`CONFIRMATION_REQUIRED`,
  `TASK_BLOCKED`, `COMPLETED`) actually flows through the wrapper's own
  control logic under a realistic retry/replay sequence.
- **E4** in particular is a genuine single-use-token security property, not
  a cosmetic idempotency detail — a stale, already-"used" token silently
  remaining valid would have let a later, unrelated confirmation decision
  be bypassed by replaying an old token.
- **E6** was a plain naming-consistency bug (no security impact) caught by
  a direct equality assertion between the receipt and the skill catalog.

## Foundation Milestone F — Finalized Decisions (Implemented)

> Records decisions made building `skilllayer.tasks.human_report` (the
> deterministic human-readable task report) on top of the unmodified
> Milestone E receipt/`public_api`/`interventions` modules.

| # | Decision | Chosen option | Alternatives | Reason | Revisit when |
|---|----------|---------------|--------------|--------|--------------|
| F1 | `vte_finalize`'s `tests_status["recorded"]` means "conclusively satisfied" (recorded AND passed), not merely "a result was reported" | `tests_satisfied = bool(tests_recorded) and tests_passed is True` is what's passed to Foundation D's `finalize_task` as `recorded`; the caller's raw `tests_recorded`/`tests_passed` signal is preserved separately as `reported_recorded`/`passed` for the receipt and report to distinguish "never ran" from "ran and failed" from "ran, outcome unknown" | Pass the caller's raw `tests_recorded` straight through as `recorded` | **Found via dogfooding, before Milestone F's report layer existed**: Foundation D's `finalize_task` only ever inspects `tests_status["recorded"]`, never `"passed"` — passing the raw flag through let `tests_recorded=True, tests_passed=False` (a real, definite test *failure*) and `tests_recorded=True, tests_passed=None` (an *inconclusive* result) both silently satisfy the "tests recorded" gate and reach `TASK_VERIFIED_COMPLETE`, which is exactly the false-completion class this whole product exists to prevent | Foundation D's `finalize_task` itself grows a native pass/fail-aware test gate |
| F2 | `overall_status` distinguishes `FAILED` from `BLOCKED` using the receipt's own `tests_summary`, not Foundation D's verdict alone | `human_report._overall_status` maps `TASK_BLOCKED` to `FAILED` specifically when `tests_summary.reported_recorded` is true and `passed` is `False`; every other `TASK_BLOCKED` reason maps to `BLOCKED` | Map `TASK_BLOCKED` to `BLOCKED` unconditionally, matching Foundation D's own vocabulary one-to-one | Foundation D's `finalize_task` collapses a definite test failure and every other block (scope/staleness/ownership) into the same `TASK_BLOCKED` verdict (per F1, it never inspects `"passed"`); a human reading "BLOCKED" for a straightforwardly failing test suite would reasonably expect "FAILED" instead — this report layer adds that distinction without changing Foundation D's own verdict vocabulary | Foundation D's `finalize_task` grows its own `TASK_TESTS_FAILED`-style verdict |
| F3 | Markdown truncation always fully preserves `## Next action` and `## Verdict`; only the sections above them are cut | The document is rendered in two halves (body, tail); if the combined length exceeds `MAX_MARKDOWN_CHARS` (8,000), only `body` is truncated (with an explicit `[Report truncated: ...]` marker) before the untouched `tail` is appended | Truncate the whole rendered string uniformly from the end | The milestone requires "exactly one next action" and a verdict in every report; truncating from the end could silently drop both on a report with many changed paths or limitations, which would make the report actively misleading rather than merely incomplete | Never |
| F4 | `write_human_report` is idempotent by exact content equality, not by existence | A retry with byte-identical `report`/`markdown` is a no-op success (`written_paths: []`, `idempotent: true`); a retry with *any* difference is rejected (`conflicting_report_rewrite`), never silently overwritten | Overwrite unconditionally on every call (latest-wins, like `result.json`) | The milestone explicitly requires "writing the same report twice must be idempotent" and "conflicting rewrites must fail explicitly" — a report is evidence derived from a specific receipt state, so a genuinely different rewrite attempt (e.g. two different processes finalizing concurrently with different test outcomes) must be surfaced, not silently resolved by last-write-wins | Never |
| F5 | `vte_status` reads (`read_human_report`) but never creates or rewrites a report | Status returns a `report_preview` only when `vte_finalize` already persisted `report.json`; the read path shares no code with the write path | Have `vte_status` build and cache a report on first call | The milestone explicitly requires "no silent report writes from read-only status operations"; a read-only tool that could write on some code paths would break the "status never mutates" guarantee every other VTE read-only tool (`get_task_status`, `assess_task_resume`) already provides and that has its own dedicated regression test | Never |
| F6 | Unsupported `locale` is rejected explicitly at both `human_report.build_human_report` and the `vte_finalize`/MCP layer, never silently substituted | `locale not in SUPPORTED_LOCALES` (currently `{"en"}`) raises `HumanReportError` in the module, and returns a structured `unsupported_locale` error one layer up | Fall back to `"en"` silently for an unrecognized locale | The milestone explicitly forbids silently using an LLM for translation and requires unsupported locales to be rejected explicitly; a silent fallback would hide a caller's mistaken assumption that translated output was returned | A translated deterministic template set for a second locale is added |

## Foundation Milestone F risks resolved during verification (not merely designed)

- **F1** was found during real dogfooding (before this report layer existed)
  by observing that a call with `tests_passed=False` still produced
  `TASK_VERIFIED_COMPLETE` — the kind of false-completion bug this entire
  product line exists to catch, found in its own finalize wrapper.
- **F2** was found while writing `TestDeterministicMapping::test_failed_tests`
  — the first implementation reported "BLOCKED" for a definite test failure,
  which is technically accurate to Foundation D's verdict but reads as
  evasive to a human expecting "FAILED" for a failing test suite.
- **F3** was found while writing the bounded-output test with 500 changed
  paths and again with oversized limitation strings — an early version
  truncated from the raw end of the string, which could (rarely, but
  possibly) cut into the required Verdict section for a sufficiently large
  report.
